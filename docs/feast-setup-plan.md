# Plan: Feast Feature Store Integration

## Context

The project computes per-team cumulative statistics (matchesPlayed, wins, draws, losses, goalsFor, goalsAgainst)
from Brazilian Série A historical data. The current pipeline writes directly to Redis via `RedisSink.kt`.

The goal is to replace the direct Redis sink with Feast so that:
- A Flink **Batch** job writes pre-computed aggregations to **Parquet** (offline store).
- Feast **materializes** Parquet → Redis (online store) for the initial historical backfill.
- A Flink **Stream** job pushes live updates to the Feast **feature server HTTP API**, which writes both Redis
  (online) and Parquet (offline) simultaneously.
- Training notebooks read historical features from the offline Parquet store via `feast.get_historical_features()`.

---

## Target Data Flow

```
── Batch backfill path ─────────────────────────────────────────────────
BRA.csv
  └─ Kafka producer (BraSerieAProducer)
       └─ Kafka topic: bra-serie-a-matches
            └─ Flink Batch job (TeamStatsBatchJob)
                 └─ Writes aggregated Parquet → data/offline_store/team_stats/
                      └─ feast materialize → Redis (online store)

── Stream path (live events) ────────────────────────────────────────────
Kafka topic: bra-serie-a-matches
  └─ Flink Stream job (VelocityFeatureJob — modified)
       └─ HTTP POST to Feast feature server (port 6566)
            ├─ Redis (online store, immediate serving)
            └─ Parquet (offline store, appended)

── Training path ────────────────────────────────────────────────────────
notebooks/bra_serie_a/bra_serie_a.ipynb
  └─ feast.get_historical_features(entity_df, features)
       └─ reads from data/offline_store/team_stats/*.parquet
```

## 2. feature_store.yaml

File: `feast/feature_store.yaml`

```yaml
project: bra_serie_a
registry: data/registry.db
provider: local

offline_store:
  type: file                           # file-based offline store backed by Parquet

online_store:
  type: redis
  connection_string: "redis:6379,db=0" # matches Docker Compose service name

entity_key_serialization_version: 3
```

Notes:
- `registry: data/registry.db` is a local SQLite registry file generated by `feast apply`. It is written into
  the `feast/data/` subdirectory (Feast resolves this path relative to `feature_store.yaml`'s location).
- `connection_string` uses the Docker Compose service name `redis` on port 6379, matching `application.yml`.
- `provider: local` is sufficient for this PoC; no cloud provider is needed.

## 3. Feature Definitions

File: `feast/features.py`

### 3a. Entity

```python
from feast import Entity, ValueType

team = Entity(
    name="team_name",
    value_type=ValueType.STRING,
    description="Brazilian Série A team identifier (lowercase, underscores)",
)
```

The entity key matches the Redis key prefix pattern already used by `RedisSink.kt`:
`team_stats:{team_name}` where `team_name` is lowercase with spaces replaced by underscores.

### 3b. PushSource

```python
from feast import PushSource, FileSource

team_stats_push_source = PushSource(
    name="team_stats_push_source",
    batch_source=FileSource(   # declared below; push source requires a batch_source reference
        path="../data/offline_store/team_stats/",
        timestamp_field="event_timestamp",
    ),
)
```

The `PushSource` is the target for the Flink stream job's HTTP POST requests. Feast routes push writes to both
the online store (Redis) and the offline store (Parquet files) automatically.

### 3c. FileSource (offline batch source)

```python
from feast import FileSource

team_stats_file_source = FileSource(
    name="team_stats_file_source",
    path="../data/offline_store/team_stats/",
    timestamp_field="event_timestamp",
)
```

### 3d. FeatureView

```python
from feast import FeatureView, Field
from feast.types import Int32
from datetime import timedelta

team_stats_fv = FeatureView(
    name="team_stats",
    entities=[team],
    ttl=timedelta(days=0),          # no expiry; stats are cumulative/historical
    schema=[
        Field(name="matches_played",  dtype=Int32),
        Field(name="wins",            dtype=Int32),
        Field(name="draws",           dtype=Int32),
        Field(name="losses",          dtype=Int32),
        Field(name="goals_for",       dtype=Int32),
        Field(name="goals_against",   dtype=Int32),
    ],
    source=team_stats_push_source,  # live stream writes come via push source
    online=True,
)
```

Field naming convention: `snake_case` to align with Python conventions. The Flink-side payload must use the
same field names when POSTing to the feature server (see Section 6).

## 4. Docker Compose Additions

Add the following service to `docker-compose.yml`:

```yaml
  feast-feature-server:
    image: feastdev/feature-server:0.40.0      # pin to a stable release
    container_name: feast-feature-server
    ports:
      - "6566:6566"                            # Feast HTTP feature server default port
    volumes:
      - ./feast:/feast                         # mounts feast/ dir with feature_store.yaml
      - ./data:/data                           # mounts data/ dir with offline store + registry
    command: >
      feast -c /feast serve
      --host 0.0.0.0
      --port 6566
    depends_on:
      - redis
    restart: unless-stopped
```

Also add a `./data:/data` bind-mount to the `flink-taskmanager` service so the Flink Batch job can write
Parquet files to the same `data/offline_store/team_stats/` directory that Feast reads from.

The `feast apply` command (Section 7) must be run before starting this container, because it creates
`data/registry.db`.

## 5. Flink Batch Job — Write Aggregations to Parquet

### 5a. New job file

Create `pipeline/src/main/kotlin/com/rtagui/jobs/TeamStatsBatchJob.kt`.

This job replaces the streaming `VelocityFeatureJob` for the initial historical backfill:

1. Source: `KafkaSource` with `OffsetsInitializer.earliest()` — consumes all 5,386 historical events from
   `bra-serie-a-matches`.
2. Transform: same `MatchToTeamRecordFlatMap` + `TeamStatsFunction` pipeline as today.
3. Sink: instead of `RedisSink`, write to a **Parquet file sink**.

### 5b. Parquet dependency

Add to `build.gradle.kts`:

```kotlin
implementation("org.apache.flink:flink-parquet:$flinkVersion")
implementation("org.apache.parquet:parquet-avro:1.14.4")
implementation("org.apache.avro:avro:1.12.0")
```

### 5c. Parquet output schema

Each Parquet row must include the following columns (matching `features.py` field names):

| Parquet column    | Type   | Source field in TeamStats  |
|-------------------|--------|----------------------------|
| `team_name`       | STRING | `teamName` (entity key)    |
| `matches_played`  | INT32  | `matchesPlayed`            |
| `wins`            | INT32  | `wins`                     |
| `draws`           | INT32  | `draws`                    |
| `losses`          | INT32  | `losses`                   |
| `goals_for`       | INT32  | `goalsFor`                 |
| `goals_against`   | INT32  | `goalsAgainst`             |
| `event_timestamp` | INT64  | epoch millis of last match |

`event_timestamp` is mandatory for Feast's offline store to locate features in time. Use the `matchTimestamp`
from the last `TeamMatchRecord` processed for each team — the `TeamStatsFunction` must be updated to propagate
this timestamp through the `TeamStats` model.

### 5d. TeamStats model change

Add `eventTimestamp: Long` to `pipeline/src/main/kotlin/com/rtagui/model/TeamStats.kt` and update
`TeamStatsFunction` to set it from `record.matchTimestamp` on each update.

### 5e. Output path

Write Parquet output to `data/offline_store/team_stats/` on the host. Configure via `AppConfig`:

```yaml
# application.yml addition
feast:
  offlineStorePath: "/data/offline_store/team_stats"
```

### 5f. Batch job entry point

Add a Gradle task in `build.gradle.kts`:

```kotlin
tasks.register<JavaExec>("runBatchJob") {
    classpath = sourceSets["main"].runtimeClasspath
    mainClass.set("com.rtagui.jobs.TeamStatsBatchJobKt")
}
```

## 6. Flink Stream Job — Replace RedisSink with Feast HTTP Push

### 6a. New sink: FeastHttpPushSink

Create `pipeline/src/main/kotlin/com/rtagui/sink/FeastHttpPushSink.kt`.

This `RichSinkFunction<TeamStats>` replaces `RedisSink` in `VelocityFeatureJob`.

Behaviour:
1. `open()`: create a `java.net.http.HttpClient` (no external dependency; Java 21 built-in).
2. `invoke(stats, ctx)`: serialize `TeamStats` to the Feast push payload JSON and POST to
   `http://feast-feature-server:6566/push`.
3. `close()`: shut down the HTTP client executor if applicable.

### 6b. Feast push payload format

The Feast feature server `/push` endpoint expects this JSON body:

```json
{
  "push_source_name": "team_stats_push_source",
  "df": {
    "team_name":       ["flamengo"],
    "matches_played":  [38],
    "wins":            [20],
    "draws":           [10],
    "losses":          [8],
    "goals_for":       [65],
    "goals_against":   [40],
    "event_timestamp": ["2024-11-10T00:00:00Z"]
  },
  "to": "online_and_offline"
}
```

Key points:
- `push_source_name` must match the `PushSource` name defined in `features.py`.
- `df` contains arrays (even for single records); the Flink sink sends one record per invocation.
- `to: "online_and_offline"` instructs Feast to write to both Redis and Parquet simultaneously.
- `team_name` must be lowercase with underscores (matching the existing key normalisation in `RedisSink.kt`).
- `event_timestamp` must be an ISO-8601 UTC string derived from `TeamStats.eventTimestamp` (epoch millis).

### 6c. VelocityFeatureJob change

In `pipeline/src/main/kotlin/com/rtagui/jobs/VelocityFeatureJob.kt`:
- Replace `RedisSink(redisUri)` with `FeastHttpPushSink(feastUrl)`.
- Remove the `redisUri` constructor parameter; add `feastUrl: String`.

### 6d. AppConfig change

In `pipeline/src/main/kotlin/com/rtagui/config/AppConfig.kt`:
- Replace `RedisConfig` with `FeastConfig(val pushUrl: String = "http://localhost:6566")`.
- Update `application.yml`:

```yaml
kafka:
  bootstrapServers: "kafka:9092"

feast:
  pushUrl: "http://feast-feature-server:6566"
  offlineStorePath: "/data/offline_store/team_stats"
```

### 6e. Main.kt change

Update `Main.kt` to pass `config.feast.pushUrl` instead of `config.redis.uri`.

## 7. feast apply + feast materialize Workflow

### 7a. Prerequisites

Install Feast on the host machine:

```bash
pip install feast==0.40.0 pyarrow pandas
```

Or add to `feast/requirements.txt`:

```
feast==0.40.0
pyarrow>=14.0
pandas>=2.0
```

### 7b. Step-by-step sequence

Execute these steps in order during initial setup:

**Step 1 — Run the Flink Batch Job (write Parquet)**

```bash
docker compose up -d kafka flink-jobmanager flink-taskmanager
./gradlew :pipeline:runProducer          # publish all 5,386 events to Kafka
./gradlew :pipeline:runBatchJob          # consume, aggregate, write Parquet to data/offline_store/
```

Verify: `data/offline_store/team_stats/` should contain at least one `.parquet` file.

**Step 2 — feast apply (register feature definitions)**

```bash
cd feast/
feast apply
```

This reads `feature_store.yaml` and `features.py`, validates the schema, and writes `data/registry.db`. It does
NOT push any data; it only registers metadata.

**Step 3 — Start infrastructure including Feast server**

```bash
docker compose up -d redis feast-feature-server
```

The feast-feature-server container loads the registry from `data/registry.db` (bind-mounted from the host).

**Step 4 — feast materialize (backfill Redis from Parquet)**

```bash
cd feast/
feast materialize 2012-01-01T00:00:00 $(date -u +"%Y-%m-%dT%H:%M:%S")
```

This reads all Parquet files from `data/offline_store/team_stats/`, computes the latest feature values per
entity, and writes them to Redis.

**Step 5 — Start stream job (live updates via HTTP push)**

```bash
./gradlew :pipeline:shadowJar
./gradlew :pipeline:deployFlink          # submits VelocityFeatureJob to Flink cluster
```

From this point forward, every new Kafka event causes Flink to POST to the Feast feature server, which updates
both Redis (online) and appends to Parquet (offline).

### 7c. materialize.sh convenience script

File: `feast/materialize.sh`

```bash
#!/usr/bin/env bash
set -euo pipefail
cd "$(dirname "$0")"
feast apply
feast materialize 2012-01-01T00:00:00 "$(date -u +"%Y-%m-%dT%H:%M:%S")"
echo "Materialization complete."
```

## 8. Training: Reading from the Offline Store

### 8a. How the notebook reads features

In `notebooks/bra_serie_a/bra_serie_a.ipynb`, replace the section that reads `BRA.csv` for feature engineering
with Feast historical lookups:

```python
import feast
from datetime import datetime
import pandas as pd

store = feast.FeatureStore(repo_path="../../feast")  # relative path from notebook to feast/

entity_df = pd.DataFrame({
    "team_name":       ["flamengo", "palmeiras"],
    "event_timestamp": [datetime(2024, 11, 10), datetime(2024, 11, 10)],
})

features = store.get_historical_features(
    entity_df=entity_df,
    features=[
        "team_stats:matches_played",
        "team_stats:wins",
        "team_stats:draws",
        "team_stats:losses",
        "team_stats:goals_for",
        "team_stats:goals_against",
    ],
).to_df()
```

`get_historical_features()` performs a **point-in-time correct join**: for each row in `entity_df`, it finds the
latest feature values where `event_timestamp <= row.event_timestamp`. This prevents data leakage.

### 8b. Building the match-level training DataFrame

A match involves two teams. Join features for both home and away teams separately:

```python
home_features = store.get_historical_features(
    entity_df=home_entity_df,    # team_name = home team, event_timestamp = match date
    features=["team_stats:wins", ...],
).to_df().add_prefix("home_")

away_features = store.get_historical_features(
    entity_df=away_entity_df,
    features=["team_stats:wins", ...],
).to_df().add_prefix("away_")

training_df = pd.concat([home_features, away_features], axis=1)
```

### 8c. Online serving (inference time)

For real-time predictions, read the latest features from Redis via:

```python
online_features = store.get_online_features(
    features=["team_stats:matches_played", "team_stats:wins", ...],
    entity_rows=[{"team_name": "flamengo"}, {"team_name": "palmeiras"}],
).to_dict()
```

## 10. Dependency Sequencing

The following order must be respected during implementation:

```
1. TeamStats.kt           — add eventTimestamp (all downstream depends on this)
2. TeamStatsFunction.kt   — propagate eventTimestamp from TeamMatchRecord
3. AppConfig / application.yml — replace RedisConfig with FeastConfig
4. features.py + feature_store.yaml — define Feast schema
5. feast apply            — must run before any Feast server starts
6. docker-compose.yml     — add feast-feature-server (needs registry.db from step 5)
7. FeastHttpPushSink.kt   — new stream sink
8. VelocityFeatureJob.kt  — swap RedisSink → FeastHttpPushSink
9. TeamStatsBatchJob.kt   — new batch job (Parquet writer)
10. feast materialize     — after batch job writes Parquet
11. Tests                 — FeastHttpPushSinkIT + updated TeamStatsFunctionTest
```

## 11. Potential Challenges

| Challenge | Mitigation |
|-----------|------------|
| Feast's Redis key format differs from current `team_stats:{name}` hash format | After migration, Redis keys use Feast's internal schema. Consumers must use `store.get_online_features()` instead of direct `redis-cli hgetall`. |
| `data/offline_store/` written by Flink (JVM) and read by Feast (Python) | Add `./data:/data` bind-mount to the `flink-taskmanager` service in `docker-compose.yml`. |
| Feast feature server startup race condition | Use `depends_on: redis` and a healthcheck; `feast serve` fails if Redis is unavailable. |
| Parquet schema evolution | Pin Avro schema version; use versioned Parquet paths if fields are added in the future. |
| HTTP POST latency from Flink to Feast server | Use `java.net.http.HttpClient` with `sendAsync` for high-throughput scenarios. Synchronous calls are acceptable for this PoC volume. |
| `feast materialize` uses wall-clock `end_date` | Use `feast materialize 2012-01-01T00:00:00 <now>` to cover the full 2012–2024 historical range. |
